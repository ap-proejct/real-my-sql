# 6. 데이터 압축 (Data Compression)

## 📌 개요

MySQL 서버 디스크에 저장된 데이터 파일의 크기는 다음과 직결됩니다:
- 쿼리 처리 성능
- 백업 및 복구 시간

많은 DBMS가 이런 문제를 해결하기 위해 **데이터 압축 기능**을 제공합니다.

---

## 6-1. 페이지 압축 (Transparent Page Compression)

### 동작 방식

**압축/해제 시점**
- **압축**: MySQL 서버가 디스크에 저장하는 시점에 데이터 페이지 압축
- **해제**: 디스크에서 데이터 페이지를 읽어올 때 압축 해제

**버퍼 풀 관리**
> 버퍼 풀에 데이터 페이지가 적재되면 **압축이 해제된 상태로만 관리**됩니다.
> 
> 압축 여부에 상관없이 투명(Transparent)하게 보입니다.

---

### 문제점과 해결 방법

#### 🔴 문제점
16KB 데이터 페이지를 압축한 결과 용량을 예측할 수 없는데, 하나의 테이블은 동일한 크기의 페이지(블록)로 통일되어야 합니다.

#### 💡 해결 방법: 펀치 홀 (Punch Hole)

**펀치 홀 동작 과정**
```
1. 16KB 페이지를 압축 (압축 결과를 7KB로 가정)

2. MySQL 서버는 디스크에 압축된 결과 7KB를 기록
   (이때 압축 데이터 7KB + 9KB의 빈 데이터를 기록)

3. 디스크에 데이터를 기록한 후, 7KB 이후의 공간 9KB에 대해 펀치 홀을 생성

4. 파일 시스템은 7KB만 남기고 나머지 디스크의 9KB 공간은 운영체제로 반납
```

#### ⚠️ 펀치 홀의 한계

- 운영체제뿐만 아니라 **하드웨어 자체에서도 해당 기능을 지원**해야 사용 가능
- 파일 시스템 관련 명령어(유틸리티)가 펀치 홀을 지원하지 못함

> 이런 이유로 실제 페이지 압축은 많이 사용되지 않는 상태입니다.

---

### 페이지 압축 설정
```sql
-- 테이블 생성 시
CREATE TABLE t1 (c1 INT) COMPRESSION="zlib";

-- 테이블 변경 시
ALTER TABLE t1 COMPRESSION="zlib";
OPTIMIZE TABLE t1;
```

---

## 6-2. 테이블 압축

### 특징

**장점**
- 운영체제나 하드웨어에 대한 제약이 없어 **활용도가 높음**

**단점**
- ❌ 버퍼 풀 공간 활용률이 낮음
- ❌ 쿼리 처리 성능이 낮음
- ❌ 빈번한 데이터 변경 시 압축률이 떨어짐

---

### 6-2-1. 압축 테이블 생성

#### 전제 조건

압축을 사용하려는 테이블이 **별도의 테이블 스페이스를 사용**해야 합니다.

#### 설정 요구사항

| 항목 | 설정 |
|---|---|
| **innodb_file_per_table** | ON |
| **ROW_FORMAT** | COMPRESSED 명시 |
| **KEY_BLOCK_SIZE** | 압축된 페이지의 타깃 크기 (2ⁿ, n≥2) |

**KEY_BLOCK_SIZE 제약사항**
- `innodb_page_size`가 16KB인 경우: 4KB 또는 8KB만 설정 가능
- 페이지 크기가 32KB, 64KB인 경우: **압축 불가**
- `KEY_BLOCK_SIZE`만 명시해도 `ROW_FORMAT=COMPRESSED`는 자동 설정
```sql
-- 예시
CREATE TABLE compressed_table (
    id INT PRIMARY KEY,
    data VARCHAR(255)
) ROW_FORMAT=COMPRESSED KEY_BLOCK_SIZE=8;
```

---

### InnoDB 압축 적용 방법
```
1. 16KB의 데이터 페이지를 압축
   
   1.1 압축된 결과가 8KB 이하
       → 그대로 디스크에 저장 (압축 완료)
   
   1.2 압축된 결과가 8KB 초과
       → 원본 페이지를 스플릿
       → 2개의 페이지에 8KB씩 저장

2. 나뉜 페이지 각각에 대해 1번 단계를 반복 실행
```

---

### 6-2-2. KEY_BLOCK_SIZE 결정

> 💡 **테이블 압축에서 가장 중요한 부분**
> 
> 압축된 결과가 어느 정도가 될지 예측해서 `KEY_BLOCK_SIZE`를 결정하는 것

#### 권장 테스트 방법

1. `KEY_BLOCK_SIZE`를 4KB 또는 8KB로 테이블 생성
2. 샘플 데이터를 저장 (최소 데이터 페이지 10개 이상)
3. 압축률 및 성능 확인
```sql
-- 테스트 테이블 생성
CREATE TABLE test_compress_4k (
    -- 컬럼 정의
) ROW_FORMAT=COMPRESSED KEY_BLOCK_SIZE=4;

CREATE TABLE test_compress_8k (
    -- 컬럼 정의
) ROW_FORMAT=COMPRESSED KEY_BLOCK_SIZE=8;
```

#### ⚠️ 주의사항

> 8KB라고 무조건 압축 성능이 더 좋거나, 그렇지 않기 때문에 **반드시 테스트가 필요**합니다.

**압축을 고려하지 말아야 할 경우**
- 테이블의 데이터가 매우 빈번하게 조회되고 변경되는 경우
- 압축 알고리즘(zlib)이 많은 CPU 자원을 소모하기 때문

---

### 6-2-3. 압축된 페이지의 버퍼 풀 적재 및 사용

#### 버퍼 풀 관리 방식

압축된 테이블의 데이터 페이지를 버퍼 풀에 적재하면 **2개 버전을 관리**합니다:

**관리되는 리스트**
- **LRU 리스트**: 디스크에서 읽은 상태 그대로의 데이터 페이지 목록
  - 압축이 적용되지 않은 테이블의 데이터 페이지
  - 압축이 적용된 테이블의 압축된 데이터 페이지

- **Unzip_LRU 리스트**: 압축된 페이지들의 압축 해제 버전

---

#### ❌ 단점

1. **메모리 낭비**
   - 압축된 테이블에 대해 버퍼 풀의 공간을 이중으로 사용

2. **CPU 소모**
   - 압축된 페이지에서 데이터를 읽거나 변경하기 위해 압축 해제 필요

---

#### 💡 Unzip_LRU 리스트를 통한 보완

요청 패턴에 따라 다음 처리를 수행합니다:

**1. 버퍼 풀 공간 확보 시**
```
LRU 리스트: 원본 데이터 페이지(압축된 형태) 유지
Unzip_LRU 리스트: 압축 해제된 버전 제거 → 공간 확보
```

**2. 압축된 데이터 페이지가 자주 사용되는 경우**
```
Unzip_LRU 리스트에 압축 해제된 페이지 유지
→ 압축 및 압축 해제 작업 최소화
```

**3. 압축된 데이터 페이지가 사용되지 않는 경우**
```
LRU 리스트에서 제거 시
→ Unzip_LRU 리스트에서도 함께 제거
```

---

#### 🔄 어댑티브 알고리즘

InnoDB 버퍼 풀에서 압축 해제된 데이터 페이지 관리를 위한 알고리즘:

| 서버 상태 | Unzip_LRU 관리 전략 |
|---|---|
| **CPU 사용량이 높은 경우** | Unzip_LRU 비율을 높여서 압축 해제 작업 최소화 |
| **Disk I/O 사용량이 높은 경우** | Unzip_LRU 비율을 낮춰서 버퍼 풀 공간 확보 |

---

### 6-2-4. 테이블 압축 관련 설정

#### 페이지 압축 실패율을 낮추기 위한 튜닝 포인트

**1. innodb_cmp_per_index_enabled**
```sql
SET GLOBAL innodb_cmp_per_index_enabled = ON;
```
- 테이블 압축이 사용된 테이블의 모든 인덱스별로 압축 성공 및 압축 실행 횟수를 수집하도록 설정

---

**2. innodb_compression_level**
```sql
SET GLOBAL innodb_compression_level = 6;  -- 기본값: 6, 범위: 0-9
```
- InnoDB의 테이블 압축은 **zlib 압축 알고리즘**만 지원
- 압축률 설정 가능
- 값이 높을수록 압축률은 높지만 CPU 사용량 증가

---

**3. innodb_compression_failure_threshold_pct / innodb_compression_pad_pct_max**
```sql
SET GLOBAL innodb_compression_failure_threshold_pct = 5;  -- 기본값: 5%
SET GLOBAL innodb_compression_pad_pct_max = 50;           -- 기본값: 50%
```

**동작 방식**
- 테이블 단위로 압축 실패율이 설정값보다 커지면
- 압축 실행 전 원본 데이터 페이지의 끝에 의도적으로 일정 크기의 **빈 공간(Padding)** 추가
- Padding의 크기는 `innodb_compression_pad_pct_max`를 초과할 수 없음

**목적**: 압축 실패를 줄여 페이지 스플릿 방지

---

**4. innodb_log_compressed_pages**
```sql
SET GLOBAL innodb_log_compressed_pages = ON;  -- 기본값: ON
```

**용도**
- MySQL 서버가 비정상적으로 종료됐다가 다시 시작되는 경우
- 압축 알고리즘의 버전 차이가 있더라도 복구 과정이 실패하지 않도록
- 압축된 데이터 페이지를 리두 로그에 저장

**주의사항**
- 리두 로그 증가로 인한 디스크 I/O 증가 가능
- 복구 안정성과 성능 사이의 트레이드오프 고려 필요

---

## 📊 압축 방식 비교

| 구분 | 페이지 압축 | 테이블 압축 |
|---|---|---|
| **투명성** | Transparent (버퍼 풀에서는 압축 해제 상태) | 압축/해제 상태 모두 관리 |
| **하드웨어 의존성** | 펀치 홀 지원 필요 | 없음 |
| **활용도** | 낮음 | 높음 |
| **설정 복잡도** | 간단 | 튜닝 포인트 多 |
| **버퍼 풀 효율** | 높음 | 낮음 (이중 관리) |
